From Multi-Echelon Inventory to Deep Reinforcement Learning

A plain-language guide to every idea behind the Geevers et al. (2024) reproduction in this repo — and exactly where each idea lives in the code.

0 · The big picture in one diagram

In one line: five ideas nested like Russian dolls — each outer layer exists only to fix a weakness of the one inside it.

1 · Multi-echelon inventory optimization

1.1 The one trade-off everything orbits

Every stocking location is caught between two costs:

  • Hold too much → you pay a holding cost h per unit each period (tied-up cash, space, spoilage).
  • Hold too little → demand shows up that you can't fill → a backorder cost b per unit each period. The unfilled demand is backlogged (remembered and filled later), not lost.

Good inventory management means sitting at the bottom of that U-shaped cost curve — without knowing future demand or exactly when shipments will land.

1.2 "Echelon" = a level in the chain

A product passes through several stocking stages before a customer buys it: factory → warehouse → depot → store. Each stage is an echelon. "Multi-echelon" means we optimise all stages together, rather than letting each location look after only itself.

Why "together" matters — the bullwhip effect. If every stage reacts only to its own incoming orders, a small wobble in store demand gets amplified into wild swings at the factory. One coordinated decision-maker who sees the whole network smooths this out. That coordinated decision-maker is exactly what our RL agent becomes.

1.3 The vocabulary to own

TermMeaning
Stock pointA place that holds inventory. In the paper: warehouses w (upstream) and retailers r (downstream, where customers buy).
On-hand inventoryUnits physically on the shelf right now.
BackordersDemand that arrived but couldn't be filled yet — owed to a customer or downstream node.
In-transit / pipelineUnits already ordered and shipped but not yet arrived, because of lead time.
Lead time LPeriods between ordering and receiving. Fixed (=1) or random (Uniform{0..4} in the beer game).
Inventory positionon-hand + in-transit − backorders. The number a good rule reacts to — it credits stock already on the way.
Base-stock (order-up-to) policy SThe classic rule: each period order max(0, S − inventory position) to refill up to a target S. Simple, often near-optimal — our benchmark.
Fill rateShare of demand filled immediately from stock. CardBoard targets 98%.

1.4 Three network shapes (each harder than the last)

ShapeDefinitionOur case
Linear / serialA single chain — each node has one supplier and one customer.Beer game: 4 nodes in a row.
DivergentOne node can supply several downstream nodes (a tree that only branches downward).1 warehouse → 3 retailers.
GeneralSome node has multiple suppliers AND some node supplies multiple customers — a real mesh.CardBoard: 4 mills, 5 plants, 14 directed edges, partly connected.

It gets harder because the number of interacting decisions explodes, and in a general network an order can be split across suppliers — a combinatorial choice on top of "how much".

1.5 The goal, precisely

minimize E[ Σt ( Σw (hw·Iw,t + bw·BOw,t) + Σr (hr·Ir,t + br·BOr,t) ) ]

Add up, over all time and all nodes, the holding cost on inventory plus the backorder cost on backlog. That single number — total expected cost — is what we drive down.

1.6 Why it's genuinely hard

  • Too many situations: the state is the inventory + pipeline + backorders of every node. With 9 nodes the number of distinct situations is astronomical.
  • Too many decisions: you set an order for every node/edge at once — a huge joint choice (numbers in §7).
  • Randomness: demand (and sometimes lead time) is uncertain, so you optimise an average, not a fixed value.

Exact optimisation can't cope; hand-built heuristics work but must be re-derived per network shape. "A general method that needs no hand-derivation per network" is the gap RL fills.

📖 Read more: Wikipedia — Bullwhip effect (5-min read on why coordination matters), and the Geevers et al. paper §3 for the exact problem set-up.

2 · Markov Decision Processes — the language

In one line: an MDP is the standard recipe-card for "make a sequence of decisions under uncertainty to optimise a long-run goal."

MDP = ( S , A , P , R , γ )
SymbolNameMeaning
SState spaceAll possible situations s.
AAction spaceAll possible decisions a.
P(s'|s,a)TransitionChance of landing in s' after doing a in s — this is where randomness lives.
R(s,a)RewardImmediate payoff. We set reward = −cost, so maximising reward = minimising cost.
γDiscountA number in [0,1) (we use 0.99) weighting future vs present, and keeping the infinite sum finite.
The Markov property — the rule that makes it an MDP. The next state and reward depend only on the current state and action, not the whole past. In practice: the state must contain everything you need to act well. If you'd want the pipeline to decide an order, the pipeline must be inside the state. Getting this right is the single most important modelling step (§3).

2.1 Policy, return, value — three words you'll reuse constantly

  • Policy π — the strategy: state → action (or a distribution over actions). Solving the MDP = finding the best policy.
  • Return Gt — the discounted total of future rewards: Gt = rt + γrt+1 + γ²rt+2 + …
  • Value Vπ(s) — expected return if you start in s and follow π. "How good is this situation?"
  • Action-value Qπ(s,a) — expected return if you do a now, then follow π. "How good is this move?"
π* = argmaxπ E[ Σt≥0 γt · rt ] (= the policy with lowest expected discounted cost)

2.2 The Bellman equation — the recursion under everything

Value is self-referential: the value of now = the reward now + the discounted value of where you land next.

Vπ(s) = E[ r + γ·Vπ(s') ] ; Q*(s,a) = E[ r + γ·maxa' Q*(s',a') ]

Classical dynamic programming solves this by sweeping the entire table of states — which is exactly what becomes impossible at scale, and why we learn instead of compute.

📖 Read more: Sutton & Barto, RL: An Introduction, Ch. 3 (MDPs) — the clearest short treatment, free online.

3 · Translating inventory INTO an MDP

In one line: pick four things — state, action, transition, reward — and the inventory problem becomes a concrete MDP any RL algorithm can attack.

3.1 State — pack in enough to stay Markov

To order well you must know what you have, what you owe, and what's coming. So the state vector glues together, across the network:

  • total inventory and total backorders (summary numbers),
  • on-hand inventory per node,
  • backorders per node/edge,
  • in-transit (pipeline) amounts — per node, and for the linear case per future arrival period (its lead time can be up to 4),
  • recent demand (linear case).
In our code. Any _obs() method in envs.py is the state vector. GeneralEnv._obs() builds [Σinv, Σbackorders, inv-per-mill, inv-per-plant, backorder-per-edge, backorder-per-plant, in-transit-per-mill, in-transit-per-edge] — the pipeline is included so the Markov property holds.

Normalization (necessary, not cosmetic)

Neural nets learn poorly when inputs have wildly different sizes. So each raw value is divided by an upper bound and squeezed into [−1,1] (or [0,1] for the general case).

obs = clip( raw_value / upper_bound , 0 , 1 ) · 2 − 1 (linear / divergent)

3.2 Action — the order quantities

Each period's decision is "how much does each node order from each supplier." The action is a vector of order quantities — one per edge (the general "order-per-edge" variant) or one per node (the simpler variant).

In our code. The action is a continuous vector in [−1,1]. _scale_action() maps each entry to [0, upper_bound] and rounds to whole units. The network outputs a number near 0; the env rescales it. Why continuous? — that's the punchline of §7.

3.3 Transition — the simulator IS P(s'|s,a)

We never write P as a formula; we simulate it. Each period a fixed run of events fires: receive arriving shipments → suppliers ship → warehouses ship to retailers (rationing when short) → place new orders → observe demand and backlog any shortfall. Randomness enters through the demand draw (and, in the linear case, the lead-time draw).

In our code. _run_period() in each env is the transition. One env.step(action) samples one s → s' and returns the reward. The agent learns P only implicitly, by living through thousands of steps.

3.4 Reward — negative, scaled cost

rewardt = − ( total holding + total backorder cost )t / 1000

Negated so a maximiser minimises cost; divided by 1000 so the critic's targets aren't huge (a training-stability trick).

3.5 Horizon & episodes

We want a stationary policy that's good "on average forever," so we train on an infinite horizon with γ=0.99 and reset the simulator every 256 steps (one PPO buffer). For evaluation we run a fixed horizon (35 / 75 / 100 periods) and throw away a warm-up so the reported cost reflects steady state.

The whole translation in one breath: state = normalised (inventory, backorders, pipeline, demand); action = scaled order-per-edge; transition = the event-sequence simulator with random demand/lead-time; reward = −cost/1000. Once this mapping exists, any RL algorithm can be pointed at it.
📖 Read more: Gymnasium — "Create a custom environment" shows the exact reset()/step()/observation_space pattern our envs.py follows.

4 · Reinforcement Learning — solving by trial

In one line: learn a good policy from experience — try, observe the reward, nudge the strategy toward what paid off — without ever being handed the answer or the equations of P.

Analogy. You're not given the recipe. You cook the dish thousands of times, taste each result (reward), and gradually adjust. RL is structured trial-and-error with memory.

4.1 Exploration vs exploitation

To find good moves you must explore (try unfamiliar actions) but eventually exploit (use what works). Too little exploring → stuck in a mediocre habit; too much → never settle. In our method, exploration comes for free: the policy is a probability distribution that starts wide and sharpens as it learns.

4.2 The three families

FamilyIdeaExamples
Value-basedLearn Q(s,a), then act greedily. The policy is implicit ("pick the highest-Q action").Q-learning, SARSA, DQN
Policy-basedParameterise the policy πθ directly and push θ uphill on reward.REINFORCE
Actor–criticBoth: an actor (policy) picks actions; a critic (value estimate) judges them to calm the actor's noisy updates.A2C, A3C, PPO

4.3 Tabular RL, and where it breaks

The original beer-game solver (Chaharsooghi 2008 — our "RLOM" benchmark) used tabular Q-learning: a giant table with one row per state and one column per action. To keep it finite they bucketed each node's inventory into 9 levels — 9⁴ ≈ 6,561 states — and limited ordering to an "X+Y" rule.

Why tables collapse. (1) Size — one cell per state×action; a general network has more than you could ever store. (2) No generalization — learning "inventory 100 is bad" teaches nothing about "inventory 101"; every cell is learned alone. (3) Forced bucketing — you crush a rich situation into coarse bins and lose information. Deep nets fix all three.

4.4 On-policy vs off-policy (one line)

Off-policy (Q-learning, DQN) can learn from old or others' data. On-policy (PPO) learns from data made by the current policy — simpler and steadier, at some sample-efficiency cost. PPO is on-policy.

📖 Read more: OpenAI Spinning Up — "Intro to RL, Part 1" (the cleanest short tour of states/actions/policies/value).

5 · Deep RL — swap the table for a network

In one line: Deep RL is just RL where a neural network stands in for the table — so you can generalise across situations and handle huge or continuous spaces.

The two superpowers a network buys you.
1. Generalization — similar states give similar outputs automatically, so experience at one situation improves behaviour at nearby ones.
2. Scale & continuity — a fixed-size network maps an arbitrarily large (even continuous) state space to outputs; you never enumerate states or actions.

5.1 What the network looks like here

Two small fully-connected nets, each 2 hidden layers × 64 units with tanh activations (the paper's choice):

  • Actor πθ(s) → a mean and spread (std) per action dimension — a Gaussian for each edge's order. The action is sampled from it, then clipped to [−1,1].
  • Critic Vφ(s) → one number, the estimated value of the state, used to judge the actor.
In our code. train.py builds exactly this with Stable-Baselines3: net_arch=dict(pi=[64,64], vf=[64,64]), activation_fn=Tanh, a Gaussian policy (the action space is a continuous Box), and glorot_init() for the paper's weight initialisation.

5.2 Policy gradients, baselines, advantage

Policy-gradient methods nudge the policy so high-return actions become more likely:

∇θ J ≈ E[ ∇θ log πθ(a|s) · A(s,a) ]

The advantage A(s,a) = Q(s,a) − V(s) asks "was this action better than the policy's average from here?" Subtracting the critic's baseline V(s) sharply cuts the noise in the gradient. The paper estimates it with GAE(λ=0.95), which smoothly trades off bias and variance.

📖 Read more: Karpathy — "Deep RL: Pong from Pixels" (famous, friendly intro to policy gradients).

6 · PPO & the continuous action space

In one line: improve the policy, but never let one update move it too far — that single rule is what makes PPO stable and forgiving.

6.1 The clipped objective

Let r(θ) = πθ(a|s) / πold(a|s) measure how much more likely the new policy makes the action. PPO maximises:

L = E[ min( r·A , clip(r, 1−ε, 1+ε)·A ) ] with ε = 0.2

If an update would change an action's probability by more than ±20%, the objective is clipped so there's no reward for pushing further — each step stays "proximal" (small). That's the whole trick behind PPO's stability, and why the paper picked it.

6.2 One training iteration

  1. Collect 256 transitions by running the current policy in the simulator.
  2. Estimate advantages (GAE) and value targets for the critic.
  3. Update for 10 passes over the buffer in minibatches of 64: actor via the clipped objective, critic via squared value error, optimiser Adam at lr 1e-4.
In our code. Every number is in the PPO(...) call in train.py: n_steps=256, batch_size=64, n_epochs=10, gamma=0.99, gae_lambda=0.95, clip_range=0.2, learning_rate=1e-4 — copied from the paper's Table 4, deliberately not tuned.

6.3 Why a Gaussian (continuous) action, not a softmax (discrete) one

A discrete policy needs one output neuron per possible action. A continuous Gaussian needs just two numbers per dimension — a mean and a spread — no matter how many quantities are possible. So the output layer grows with the number of edges, not the number of order combinations. That difference is the linchpin — quantified next.

📖 Read more: Spinning Up — PPO (intuition + pseudocode), or the original PPO paper (arXiv:1707.06347).

7 · Why DRL (and continuous actions) for THIS problem

7.1 The action-space explosion — the decisive argument

Say each node orders an integer in [0,30] — 31 choices per node. A discrete action head needs one output neuron for every joint combination:

NetworkDecision dimsDiscrete joint actionsContinuous outputs needed
Linear4 nodes31⁴ ≈ 9.2 × 10⁵8 (mean+std × 4)
General (CardBoard)18 edges31¹⁸ ≈ 7 × 10²⁶36 (mean+std × 18)

A discrete output layer for the general case can't even be allocated in memory. A continuous Gaussian head needs ~36 numbers. That is why the paper — and our reproduction — use a continuous action space with DRL. It's the only thing that scales to a general network.

7.2 The reasons, stacked

NeedTabular RLDeep RL (ours)
Huge / continuous stateMust bucket; loses info; table too bigNetwork reads the raw normalised vector
Huge action spaceOne column per action — impossibleOutputs distribution params; grows linearly
GeneralizationNone — every cell independentBuilt in — similar states → similar actions
General networks with no re-derivationn/aSame algorithm, just swap the env

7.3 DRL vs the classical alternatives

  • Exact dynamic programming: optimal in theory, intractable beyond toy sizes (the curses of dimensionality).
  • Heuristics (base-stock): fast and often near-optimal, but need problem-specific derivation (e.g. Rong et al.'s only fits divergent networks). They're our benchmarks, not a general solver.
  • DRL: one method, few assumptions, works on linear/divergent/general by swapping the environment. That flexibility is the paper's thesis — and our reproduction confirms it beats the benchmark on all three (while honestly flagging where the paper's stability story didn't replicate; see REPORT.md).
📖 Read more: Geevers et al. §4.4 (discrete vs continuous actions), and Dulac-Arnold et al., "Challenges of Real-World RL" for why large action spaces are hard.

9 · The single thread, end to end

  1. A company holds stock at many linked locations and wants minimum total holding+backorder cost under random demand. (§1)
  2. That's a sequence of decisions under uncertainty → write it as an MDP. (§2)
  3. Make it concrete: state = normalised inventory/pipeline/backorders, action = scaled order-per-edge, transition = event simulator, reward = −cost/1000. (§3)
  4. Solving the MDP exactly is impossible, so learn a policy from simulated experience → RL. Tables can't handle the size/continuity. (§4)
  5. Put the policy & value in neural networks → Deep RL: generalisation + scale, continuous actions via a Gaussian head. (§5)
  6. Use PPO — stable actor–critic with a clipped update — because the general network's discrete action set (31¹⁸) is impossible while its continuous head is tiny. (§6–7)
  7. Implement across linear / divergent / general, run 10 seeds, compare to base-stock benchmarks, report honestly. (§8, REPORT.md)

Glossary (quick recall)

Echelon One stocking stage in the chain. Multi-echelon = optimise all stages jointly.
Holding / backorder cost Per-unit-per-period penalty for stock you hold / demand you can't meet.
Inventory position on-hand + in-transit − backorders; what a base-stock rule tops up.
Base-stock policy (S) Order up to a fixed target each period; the heuristic benchmark.
Fill rate Share of demand served immediately from stock (CardBoard target = 98%).
MDP (S, A, P, R, γ) — the formal frame for sequential decisions under uncertainty.
Markov property Future depends only on current state+action ⇒ the state must hold all decision-relevant info.
Policy π The strategy: state → action (or a distribution over actions).
Value V(s) / Q(s,a) Expected discounted future reward from a state / state-action.
Bellman equation value(now) = reward + γ·value(next); the recursion DP/RL exploit.
Reinforcement learning Learn a policy from sampled interaction, no closed-form model needed.
Exploration vs exploitation Try new actions vs use known-good ones.
Tabular RL Store value in a table; breaks on large/continuous spaces, no generalisation.
Deep RL / function approximation Use a neural net instead of a table → generalise & scale.
Actor–critic Actor = policy, critic = value baseline that lowers gradient noise.
Advantage A(s,a) Q(s,a) − V(s): was this action better than average? GAE estimates it.
Policy gradient Push policy params to make high-advantage actions more probable.
PPO Policy gradient with a clipped objective so each update stays small & stable.
Continuous action space Output a Gaussian (mean,std) per dimension; scales linearly with #edges vs exponentially for discrete.
γ (discount) Weights future vs present reward; we use 0.99.
